# paseo-vscode 需求与设计

## 目标

- 为 [getpaseo/paseo](https://github.com/getpaseo/paseo) 提供 VS Code/code-server 扩展。
- 用户打开文件夹后，可以在 VS Code 侧边栏连接本地 paseo daemon，并与当前文件夹相关的多个 agent 会话聊天。
- 交互体验接近 Codex VS Code 扩展和 Paseo Web UI 的组合：用户可以创建、切换、继续多个聊天窗口。
- 交互体验需要接近 Codex 插件：以“任务列表 + 当前任务线程 + 底部 composer”为主，而不是传统表单式聊天。
- 视觉细节需要对齐 Codex 扩展：界面保持紧凑，composer、任务列表、线程消息、工具详情、按钮和菜单使用细边框、合适圆角、精简留白，并具备清晰 hover/focus 状态。
- 与 Codex 插件的核心差异是后端使用 paseo daemon；VS Code/code-server 关闭后，任务仍由 daemon 持续运行。
- 首版优先支持 code-server 使用场景，并通过 Docker E2E 保证扩展、daemon 和 UI 在 code-server 中可用。

## 首版范围

- 仅支持打开文件夹场景；无 folder 时展示不可用状态。
- 多根工作区首版只取第一个 workspace folder，作为所有 agent 的 `cwd`。
- 一个聊天窗口对应一个 Paseo agent session；首版不实现 Paseo chat room。
- UI 使用 VS Code Activity Bar + WebviewView，自研 Codex 风格任务列表、消息流、底部 composer、provider/model/mode 选择。
- 首版不做语音、relay 配对、文件/图片附件、终端流、真实 provider 凭据管理。
- daemon 由扩展负责检测、连接和必要时启动；VS Code/code-server 退出后 daemon 继续后台运行。

## 用户流程

1. 用户在 VS Code 或 code-server 中打开一个文件夹。
2. 用户打开 Paseo 侧边栏。
3. 扩展尝试连接本地 paseo daemon。
4. 如果 daemon 不存在或不可达，扩展使用 VSIX 内置运行时离线启动 daemon。
5. 扩展连接 daemon 后，拉取当前文件夹相关的 agent 列表。
6. 用户在底部 composer 输入消息；无当前任务时创建新 agent，有当前任务时继续发送到当前 agent。
7. 用户可以在任务列表和当前任务线程之间切换，查看历史 timeline，并继续发送消息。
8. 用户关闭 VS Code/code-server 后，daemon 保持后台运行；下次打开扩展时复用已有 daemon。

## 用户视角验收补充

- 需要以首次安装用户的视角打开扩展，覆盖 daemon 自动连接、provider 默认选择、首次发送、任务线程展示和返回任务列表路径。
- provider 快照晚于 Webview 初始化返回时，composer 必须自动选中当前可用的默认 provider，用户不应因为未手动切换 provider 导致首次发送失败。
- 命令面板创建新任务时必须复用当前 composer 默认 provider/model/mode 规则，不能走到不可用 provider。
- 用户未先打开 Paseo 侧栏时，也可以直接通过命令面板创建新任务，命令入口需要自行启动并连接 daemon。
- 发现影响首次使用、消息发送、任务切换或错误重试的问题后，需要补充 E2E 用例并修复。
- 线程输出需要对齐 Paseo Web UI / Codex 体验：流式 assistant token 必须合并为连续回复，reasoning/tool/todo 作为紧凑处理事件展示，`turn_started`、`turn_completed` 等生命周期事件不能直接显示成系统事件。
- 工具调用在线程中必须以可折叠详情展示；折叠标题要清楚显示工具类型和关键信息，Shell 工具标题需要显示实际 shell 命令，展开后能看到该工具调用输出详情；其他工具调用也需要有可读标题和可展开详情。
- assistant 输出需要按 Markdown 渲染，至少支持段落、列表、代码块、行内代码和链接；Webview 不依赖远端 CDN，不执行不可信 HTML 或脚本。
- 底部 composer 需要接近截图形态：整体是大圆角输入区，顶部为多行输入和 placeholder，底部左侧只常驻加号/附件入口，右侧放 codex/claude 等 provider、模型、思考强度和发送按钮；“计划模式”和“IDE 背景信息”只保留在附加菜单中，不作为常驻入口。
- 顶部、任务列表、线程和 composer 的设置、刷新、新任务、返回、发送、停止、加号、归档、复制等关键按钮使用本地 Codex/codicon 风格图标，并保留清晰的 `title` 和 `aria-label`。
- agent/provider/model/mode 的选择与展示不使用 `-` 作为缺省、拼接或占位；缺失 model/mode 时展示自然中文占位或隐藏该段信息。
- 任务列表不展示搜索框，也不展示“全部/运行中/已归档”过滤控件；始终显示当前 workspace 的全部任务。
- 顶部、任务列表和线程页不展示运行中任务数量徽标，也不出现“正在进行中”计数文案。
- 开发流程按“发现问题、补测试、修复、验证、提交”的循环持续推进。

## Codex 风格体验

- 首屏是“任务”视图：
  - 顶部展示刷新、设置、新任务入口和 daemon 状态，不展示运行中任务数量。
  - 中部展示当前 workspace 的全部任务列表，不提供搜索或状态过滤。
  - 任务项展示标题、provider、model（如有）、状态、更新时间；未归档任务提供归档入口。
  - 底部固定 composer。
- 当前任务线程：
  - 顶部展示返回、任务标题和操作入口，不展示运行中任务数量。
  - timeline 展示用户消息、assistant 回复、处理中分组、工具调用、todo 和错误。
  - 处理中和工具调用默认可折叠，展示状态摘要。
  - 运行中时 composer 发送按钮变成停止按钮，并调用 daemon 停止当前 agent。
- Composer：
  - composer 使用大圆角输入区，顶部展示多行输入，底部展示操作控件。
  - 左侧只常驻加号/附件入口，右侧展示 provider/model/mode 选择和发送/停止按钮。
  - provider 选择作为 codex/claude 等 provider 的切换入口。
  - 新任务发送时使用当前 provider/model/mode 创建 agent。
  - 当前任务中允许切换同 provider 的 model/mode，影响后续消息。
  - “添加附件和上下文”菜单包含计划模式、IDE 背景信息和当前文件/选区入口。
  - IDE 背景信息使用 text attachment 传给 daemon，包含 workspace、当前文件和选区或可见范围摘要。
  - 计划模式优先使用 provider 原生模式；没有原生模式时在消息前追加只规划、不改文件的约束。
- 设置入口参考 Codex 设置体验，首版打开 VS Code 设置页中的 Paseo 配置。

## Daemon 管理

- 扩展激活后先尝试连接本地 daemon，不直接启动新 daemon。
- 连接目标按以下优先级推导：
  - 用户配置的 daemon host。
  - `PASEO_HOST` 环境变量。
  - `PASEO_HOME/paseo.pid` 中记录的 listen/socket 信息。
  - paseo 配置文件中的 listen 信息。
  - 默认 `localhost:6767`。
- daemon 不可达时，扩展使用 VSIX 内置依赖启动 `@getpaseo/server` daemon。
- daemon 必须 detached 后台运行；扩展 `deactivate` 只断开 WebSocket client，不停止 daemon。
- daemon 启动时禁止运行 `npm install`、`curl`、访问 npm registry 或下载远端资源。
- VSIX 必须包含启动 daemon 所需的生产运行时依赖，避免运行阶段补装依赖。
- daemon 开启密码认证时，扩展需要支持 `paseo.daemon.password`、`PASEO_PASSWORD` 和 `tcp://...?password=...` 三种传入方式。
- 本地排障时，用户可以通过 `Paseo: Open Logs` 命令打开扩展 Output 和 daemon 日志。
- code-server E2E 中需要设置：
  - `PASEO_NODE_ENV=development`，启用 paseo `mock` provider。
  - `PASEO_PASSWORD`，覆盖 password-protected daemon 连接路径。
  - `PASEO_RELAY_ENABLED=0`，避免 daemon 连接 relay。
  - `PASEO_DICTATION_ENABLED=0` 和 `PASEO_VOICE_MODE_ENABLED=0`，避免语音模型下载。

## 架构设计

### Extension Host 层

- 负责 VS Code 扩展生命周期、命令注册、WebviewViewProvider 注册和 workspace folder 解析。
- 对外暴露命令：
  - `paseo.newAgent`：创建当前文件夹的新 agent。
  - `paseo.refresh`：刷新 daemon 状态、provider 状态、agent 列表和当前 timeline。
  - `paseo.reconnectDaemon`：主动重连 daemon。
  - `paseo.showDaemonStatus`：显示 daemon host、pid、日志路径和连接状态。
  - `paseo.openLogs`：打开 Output 面板中的 Paseo 扩展日志，并尝试打开 daemon 日志文件。
- 负责把 Webview 消息转换为 Paseo Client 层调用，并把状态更新推送回 Webview。

### Paseo Client 层

- 封装 `@getpaseo/server` 暴露的 `DaemonClient`。
- 负责连接、重连、关闭、RPC 调用、事件订阅和错误归一化。
- daemon 连接传输、协议版本兼容、Webview 状态映射应独立成模块，避免服务类继续膨胀。
- 需要支持以下能力：
  - 拉取 agent 列表。
  - 创建 agent。
  - 加载 agent timeline。
  - 发送 agent 消息。
  - 拉取 provider/model/mode 状态。
  - 订阅 `agent_stream` 和 `agent_update`。

### State 层

- 维护扩展侧状态：
  - 当前 workspace folder。
  - daemon 连接状态。
  - provider/model/mode 快照。
  - 当前文件夹相关 agent 列表。
  - 当前选中的 agent。
  - 当前 agent 的 timeline。
  - 最近一次错误和可重试动作。
- agent 列表必须按当前 `cwd` 过滤，避免展示其他目录的会话。

### Webview 层

- 负责渲染 Paseo 侧边栏 UI，通过 `postMessage` 与 Extension Host 通信。
- 渲染代码按顶部/任务列表/线程/composer/DOM helper/样式拆分，避免单个 Webview 文件承载全部交互。
- UI 分区：
  - 顶部状态栏：daemon 状态、当前 folder、刷新、设置和新任务入口。
  - 任务列表：名称、provider、model（如有）、状态、更新时间和归档入口，始终显示全部任务。
  - 底部 composer：大圆角输入区、附件/上下文菜单、计划模式、provider/model/mode 选择和发送/停止按钮。
  - timeline 区域：展示 user、Markdown assistant、reasoning、tool call、todo、error 和处理中分组。
  - 发送中禁用重复提交；当前 agent 运行中时允许停止。
- Webview 需要避免依赖远端 CDN；所有脚本、样式和图标资源随扩展打包。

### E2E 层

- 通过 Docker 构建 VSIX、安装到 code-server，并用 Playwright 驱动浏览器测试。
- `go-task e2e -- --case <name>` 是统一入口。
- E2E runner 使用 Go 编写；如需浏览器自动化，使用 `playwright-go`。
- E2E 代码按编排、runner、用例断言、等待工具和通用支持拆分，便于新增 case。

## Paseo API 使用

- `fetchAgents`：拉取 agent 列表，并按 `agent.cwd === 当前 workspace folder` 过滤。
- `createAgent`：创建 agent，`config.cwd` 固定为当前首个 workspace folder。
- `fetchAgentTimeline`：加载当前 agent 的历史消息。
- `sendAgentMessage`：发送后续用户消息。
- `getProvidersSnapshot`：获取 provider、model、mode、可用性和错误状态。
- `agent_stream`：实时追加 timeline 事件。
- `agent_update`：更新 agent 状态、标题、权限、更新时间等元信息。

## UI 行为

- 无打开文件夹时：
  - 展示“需要先打开文件夹”的状态。
  - 不连接 daemon，不创建 agent。
- daemon 连接中：
  - 展示连接状态和当前连接目标。
  - 禁用创建 agent 和发送消息。
- daemon 不存在时：
  - 展示“正在启动 Paseo daemon”。
  - 启动成功后自动连接并刷新列表。
  - 启动失败时展示日志路径和错误摘要。
- agent 列表：
  - 默认按最近更新时间倒序。
  - 显示运行中、空闲、错误、等待权限等状态。
  - 不提供搜索框或全部/运行中/已归档过滤控件。
  - 选中 agent 后加载并展示 timeline。
- 新建 agent：
  - 默认 provider 优先选择可用且有默认模型的 provider。
  - 在 E2E 中使用 `mock` provider。
  - 在本地 code-server 中允许用户选择真实可用的 `claude` 或 `codex` provider，并与 AI 进行真实对话。
  - 创建时传入当前 workspace folder 作为 `cwd`。
- 发送消息：
  - 输入为空时不可发送。
  - 发送中禁用重复提交。
  - 失败时保留输入内容并展示错误，允许用户重试。

## code-server E2E 测试设计

### 总体流程

1. `go-task e2e -- --case <name>` 启动 Go e2e orchestrator。
2. orchestrator 清理上一次测试容器、网络和临时目录。
3. Docker 构建 code-server 测试镜像。
4. 镜像中构建 VSIX，并通过本地 VSIX 安装扩展。
5. Docker Compose 启动 code-server 容器和 Playwright runner 容器。
6. runner 使用 Playwright 打开 code-server，进入 Paseo 侧边栏并执行测试。
7. 测试失败时收集 code-server 日志、daemon 日志和截图，便于定位问题。
8. orchestrator 停止并清理容器。

### Docker 构建

- code-server 镜像基于固定版本 code-server 官方镜像。
- 在 code-server 基础镜像中执行扩展构建，保证 VSIX 内 native dependency 与运行时环境一致。
- 构建阶段允许联网下载 npm 包、Docker 基础镜像和 Playwright 浏览器。
- 运行阶段不允许访问外网。
- VSIX 安装方式为本地文件安装：
  - `code-server --install-extension /tmp/paseo-vscode.vsix`
- 运行时不能通过扩展市场、npm registry 或 curl 获取任何扩展依赖。

### Docker Compose 网络

- Compose 使用至少两个服务：
  - `code-server`：运行安装了 paseo-vscode 的 code-server。
  - `e2e-runner`：运行 Go e2e 和 Playwright。
- 两个服务加入同一个 Docker internal network。
- internal network 只允许容器之间互通，不提供外网出口。
- code-server 默认不暴露宿主机端口；调试任务可单独开启端口映射。

### code-server 环境变量

- `PASSWORD=paseo-e2e`：固定测试登录密码。
- `PASEO_HOME=/tmp/paseo-home`：隔离测试数据。
- `PASEO_NODE_ENV=development`：启用 `mock` provider。
- `PASEO_RELAY_ENABLED=0`：禁用 relay。
- `PASEO_DICTATION_ENABLED=0`：禁用 dictation。
- `PASEO_VOICE_MODE_ENABLED=0`：禁用 voice mode。
- `PASEO_PASSWORD=paseo-e2e-daemon`：开启 daemon 密码认证，并验证扩展会携带密码连接。
- `PASEO_LISTEN=127.0.0.1:6767`：固定 daemon 监听目标，便于断言。

### 离线约束断言

- 运行阶段网络为 internal network，容器不能访问外网。
- daemon 启动日志中不能出现 npm install、registry、curl、relay 连接、语音模型下载等行为。
- 扩展安装目录中应包含 daemon 所需生产依赖。
- `offline-daemon-start` 用例需要在 daemon 不存在时通过扩展启动 daemon，并成功连接。

## E2E 用例

### offline-daemon-start

- 启动前确保 `PASEO_HOME` 为空。
- 打开 code-server 和 Paseo view。
- 断言 UI 从“连接中/启动中”进入“已连接”。
- 断言 `PASEO_HOME/paseo.pid` 存在。
- 断言 daemon listen 为 `127.0.0.1:6767` 或扩展配置中指定值。
- 断言 daemon 日志没有外网下载或安装行为。

### mock-chat

- 使用底部 composer 选择 `mock` provider 并创建新 agent。
- 初始 prompt 使用稳定文本，例如“请输出一段测试消息”。
- 断言 agent 出现在当前文件夹列表中。
- 断言 timeline 出现用户消息。
- 断言 assistant stream 有内容追加。
- 发送第二条消息，断言后续消息进入同一个 agent timeline。

### codex-like-ux

- 打开 Paseo view。
- 断言展示任务视图、运行中任务计数和底部 composer。
- 使用 `mock` provider、`ten-second-stream` model、`load-test` mode 创建新任务。
- 打开 composer 菜单，断言 IDE 背景信息和计划模式入口存在。
- 断言创建后进入线程页，出现处理中分组、停止按钮和 assistant 回复。
- 返回任务列表，断言任务项和归档入口存在。

### reload-reconnect

- 在 `mock-chat` 或 `codex-like-ux` 创建 agent 后刷新 code-server 页面。
- 重新打开 Paseo view。
- 断言扩展自动重连 daemon。
- 断言已有 agent 仍在列表中。
- 断言当前 agent timeline 可重新加载。

### no-folder

- 启动没有 workspace folder 的 code-server。
- 打开 Paseo view。
- 断言 UI 展示“需要先打开文件夹”状态。
- 断言不会创建 agent。
- 断言不会因为缺少 folder 崩溃。

## 本地测试入口

- `npm run compile`：TypeScript 编译。
- `go-task install`：编译并打包 VSIX，然后安装到宿主机 code-server 和 vscode-server。
- `npm test`：单元测试。
- `go-task e2e -- --case offline-daemon-start`：单个 E2E 用例。
- `go-task e2e`：完整 E2E。
- 本地 code-server 验收：安装 VSIX 后打开 Paseo view，确认 daemon 状态为“已连接”，provider 快照中 `claude` 或 `codex` 为 ready，并创建一个真实 agent 验证 assistant 回复。
- 如果修改代码并新增需求，需要先补对应 E2E 用例，确认失败后再实现，最后跑全量 E2E。

## 验收标准

- `npm run compile` 通过。
- 单元测试通过。
- `go-task e2e` 全部通过。
- E2E 运行阶段断网，daemon 仍能由扩展启动。
- 扩展启动 daemon 时不访问 npm registry、不运行安装命令、不下载远端资源。
- VS Code/code-server 退出后，paseo daemon 仍在后台运行。
- 本地 code-server 中 `codex` 或 `claude` ready 时，可以创建真实 agent 并收到 assistant 回复。
- 无 folder、多根工作区首个 folder、daemon 不存在、daemon 已存在、daemon 重连等路径行为明确且稳定。
- 大文件需要持续拆分，核心业务、视图映射、UI 渲染、测试编排各自保持清晰边界。

## 风险与约束

- `@getpaseo/server` 可能包含 native dependency；VSIX 构建环境需要与 code-server 运行环境一致。
- paseo 上游启动入口、mock provider 或环境变量可能变化；实现时若发现变化，需要同步更新本文档。
- code-server UI 结构可能随版本变化；E2E 应固定 code-server 镜像版本，避免选择器频繁失效。
- 真实 provider 需要外部凭据和网络，不纳入默认 E2E。
